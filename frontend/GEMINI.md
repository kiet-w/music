# Music Player Frontend (GEMINI.md)

This file provides instructional context and mandates for AI agents working within the `frontend` directory of the Music Player project.

## Project Overview
A mobile-first music application built with Next.js and wrapped with Capacitor for cross-platform distribution (Android/iOS). It features seamless audio playback, YouTube music conversion, and Google Drive integration.

- **Primary Goal:** Provide a polished, offline-capable music playback experience.
- **Key Features:**
  - YouTube URL conversion to MP3 (via NestJS backend + `yt-dlp`).
  - Google Drive music file importing.
  - Album-based library management.
  - Internationalization support (English, Vietnamese).
  - Real-time synchronization (via Supabase).

## Tech Stack
### Core
- **Framework:** Next.js 14 (App Router)
- **Language:** TypeScript
- **Styling:** Tailwind CSS + shadcn/ui
- **State Management:** Zustand
- **Audio Engine:** Howler.js (with HTML5 fallback)
- **Internationalization:** next-intl

### Mobile & Native
- **Wrapper:** Capacitor
- **Native APIs:** @capacitor/filesystem (for offline storage)

### Infrastructure
- **Backend Communication:** Standard `fetch` API via `src/lib/api.ts`.
- **Real-time/DB:** Supabase (Client-side usage for realtime updates).

## Development Conventions
### 1. Mobile-First Design
- Always prioritize the mobile user experience. Use responsive Tailwind classes (`sm:`, `md:`, etc.) only when necessary for desktop optimization.
- Use **shadcn/ui** components for consistent UI elements.

### 2. State Management
- **Zustand Stores:**
  - `usePlayerStore`: Manages audio playback state (current track, isPlaying, volume, seek).
  - `useAuthStore`: Manages user session and tokens.
  - `useAlbumStore`: Manages library data and caching.
- Avoid local state for data that needs to persist across pages (e.g., player state).

### 3. Audio Playback
- Use `Howler.js` via `usePlayerStore`.
- Ensure `html5: true` is set in Howl instances to support large files and mobile background playback.

### 4. Internationalization (i18n)
- All user-facing text must be translated via `next-intl`.
- Translation files are located in `src/messages/en.json` and `src/messages/vi.json`.
- Components should use the `useTranslations` hook.

### 5. API Interaction
- Centralize all backend calls in `src/lib/api.ts`.
- Use the `getAuthHeaders(appToken)` helper for authenticated requests.
- Handle errors gracefully and provide user feedback via UI toasts or alerts.

## Project Structure
- `src/app/[locale]/`: Main App Router pages with i18n routing.
- `src/components/`:
  - `atoms/`: Basic UI components (mostly shadcn).
  - `molecules/`: More complex, reusable UI patterns (PlayerBar, Navbar, Library).
  - `templates/`: Full-page layouts/client components.
- `src/hooks/`: Custom React hooks (e.g., `useGoogleDrive`, `useOfflineStorage`).
- `src/store/`: Zustand state definitions.
- `src/lib/`: Utilities, API client, and Supabase config.

## Environment Variables
Ensure these are set in your `.env.local`:
- `NEXT_PUBLIC_API_URL`: Backend API URL (default: http://localhost:3002).
- `NEXT_PUBLIC_SUPABASE_URL`: Supabase project URL.
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`: Supabase anonymous API key.
